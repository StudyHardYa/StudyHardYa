远程调用：

RestTemplate：

缺陷：硬编码，不好维护，修改，远程调用方如果是集群，硬编码不可访问多个

注册：

/**
 * restTemplate 用于远程调用
 * @return
 */
@Bean
public RestTemplate restTemplate(){
    return new RestTemplate();
}

调用远程：

@Autowired
private OrderMapper orderMapper;

@Autowired
private RestTemplate restTemplate;

public Order queryOrderById(Long orderId) {
    // 1.查询订单
    Order order = orderMapper.findById(orderId);
    /**
     * 远程调用user的方法，查询user信息
     * get请求使用getForObject
     * post请求使用postForObject
     */
    String url="http://localhost:8081/user/"+order.getUserId();
    //发送远程
    User user = restTemplate.getForObject(url, User.class);
    order.setUser(user);
    // 4.返回
    return order;
}

Eureka注册中心：

流程图与介绍：

过程讲解：

​ eureka客户端将服务信息注册到eureka服务的（也就是注册中心），注册后eureka会将信息记录下来，当消费者需要使用时会向eureka注册中心去获取，如果有则会将所有需要（所有的意思：因为会有集群）的返回给消费者，消费者只需要使用负载均衡进行去访问即可，为了保证集群所注册的服务状态是安全的，Eureka有一个心跳机制，每30秒需要服务提供者发送一次消息，如果没发送则会将该服务信息给移出。

在Eureka架构中，微服务角色有两类:
1、EurekaServer:服务端，注册中心
记录服务信息
心跳监控
2、EurekaClient:客户端
Provider:服务提供者，例如案例中的user-service
注册自己的信息到EurekaServer
每隔30秒向EurekaServer发送心跳
consumer:服务消费者，例如案例中的order-service
根据服务名称从EurekaServer拉取服务列表
基于服务列表做负载均衡，选中一个微服务后发起远程调用

访问地址：

这里我们将ip地址与端口都已eureka的服务名称形式访问，由eureka进行分发（这里实际上是Ribbon进行拦截获取地址的），此别名配置的访问名称，大小写不敏感

更改策略：

 * 配置负载均衡策略

 * 优先级高于yaml配置

方法二：

加入yaml配置，可以指定哪个服务使用什么策略

eureka:
  client:
    service-url:
      defaultZone: http://127.0.0.1:9001/eureka #要注册到的eureka地址

userservice: #指定的服务名称
  ribbon:
    NFLoadBalancerRuleClassName: com.netflix.loadbalancer.RoundRobinRule #采用的负载均衡策略

开启饥饿加载：

Ribbon默认采用懒加载，即第一次访问才会去创建LoadBalanceClient，请求时间会很长。

而饥饿加载则会在项目启动时创建，降低第一次访问的耗时。

未开启饥饿加载：

在我们访问服务时，他没有采用饥饿加载会当第一次访问时才进行加载，访问时间会长一些

开启饥饿加载后：

可以发现时间减少了很多，但还要这么长时间是因为还要其他的东西要加载，如spring bean的加载，这些也都是可以配置的

开启方式：

ribbon:
  eager-load:
    enabled: true #开启饥饿加载
    clients: #指定对userservice这个服务饥饿加载
      - userservice

单个服务可以这样配置：

ribbon:
  eager-load:
    enabled: true #开启饥饿加载
    clients: userservice #指定对userservice这个服务饥饿加载

Nacos负载均衡策略：

我们配置了服务分级后，可以让同集群的获取服务优先访问同集群的，如果获取同集群失败则再去获取其他集群的

不使用这个策略会全部集群服务都访问

NacosRule:

这个策略会优先访问同集群的服务，如果没有再去服务别的集群服务，访问别的集群服务时会打印一个警告日志，该策略采用随机访问

UserService: #指定的服务名称
  ribbon:	#采用的负载均衡策略，由于nacos要优先访问同集群下的服务，我们要改为alibaba的负载均衡策略
    NFLoadBalancerRuleClassName: com.alibaba.cloud.nacos.ribbon.NacosRule

Nacos权重：

实际部署中会出现这样的场景:
服务器设备性能有差异，部分实例所在机器性能较好，另一些较差，我们希望性能好的机器承担更多的用户请求
Nacos提供了权重配置来控制访问频率，权重越大则访问频率越高

使用：

权重范围：0~1

权重越小范围的数量也就会跟少

权重为0时将不会访问该服务实例

Nacos流程图：

与Eureka区别：

1、

主要在于服务提供者有着临时实例与非临时实例

临时实例：临时实例就与Eureka中存储的实例一样，采用心跳监测，当宕机时会将该临时实例去除，采用的频率时间不一样

非临时实例：该实例不再使用心跳监测，不需要实例方式发送给Nacos告诉他自己是否存活，而是由Nacos主动询问，如果宕机也不会去除掉该服务实例，而是显示不可以，等服务实例从新启动。

2、

Eureka它是消费者每间隔30会向Eureka获取服务实例，比较麻烦，并且中间的时间如果有实例宕机了，没有感知，不能立刻向Eureka从新获取

消费者向nacos获取服务实例，会将获取服务实例，如何缓存在本地，如果nacos发现服务实例宕机了或者其他的会主动推送更变消息给我们的消费者

3、

  1. Nacos与eureka的共同点
    都支持服务注册和服务拉取
    都支持服务提供者心跳方式做健康检测
  2. Nacos与Eureka的区别
    Nacos支持服务端主动检测提供者状态:临时实例采用心跳模式，非临时实例采用主动检测模式临时实例心跳不正常会被剔除，非临时实例则不会被剔除
    Nacos支持服务列表变更的消息推送模式，服务列表更新更及时
    Nacos集群默认采用AP方式，当集群中存在非临时实例时，采用CP模式;Eureka采用AP方式

配置热部署：

什么是热部署，热部署就是在项目已启动情况下修改配置，即使不重启依旧可以获取到最新的配置

Nacos热部署，有两种方式：

方式一：

​ 使用@Value 加 @RefreshScop 两给注解进行完成热部署

​ 当配置更新后value获取到的也会是最新的

@Slf4j
@RestController
@RequestMapping("/user")
@RefreshScope
public class UserController {

    @Value("${pattern.dateformat}")
    private String dateformat;

    @GetMapping("now")
    public String now(){
        System.out.println(dateformat);
        return LocalDateTime.now().format(DateTimeFormatter.ofPattern(dateformat));
    }

}

方式二：

​ 使用@ConfigurationProperties注解进行完成

​ 只需要注入这个对象就可以使用，里面的值也是热部署的，@ConfigurationProperties(prefix = “pattern”) 获取pattern的节点，于属性名称拼接相同的就会 注入

@Data
@ToString
@Component
@ConfigurationProperties(prefix = "pattern")
public class Configproperties {

    private String dateformat;
}

Global全局过滤器：

全局过滤器的作用也是处理一切进入网关的请求和微服务响应，与GatewayFilter的作用一样。

区别在于GatewayFilter通过配置定义，处理逻辑固定。而GlobalFilter的逻辑需要自己写代码实现。

定义方式是实现GlobalFilter接口。

/**
*
*/
public interface GlobalFilter {
    /**
    * exchange 请求上下文，里面可以获取Request、response等信息
    * chain 用来把请求委托给下一个过滤器
    * return 返回表示单前过滤器业务结束
    */
    Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain);
}

使用：

一定要加上@Component让此类注册到bean中与（Order()注解或Ordered接口）

使用注解Order或实现Ordered接口都可以实现设置过滤器优先级

//@Order(-1) 用于设置过滤器优先级，数字越大优先级越小，默认2147483647最小 做大-2147483647
//@Order(-1)
@Component
public class AuthorizeFilter implements GlobalFilter , Ordered {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        //获取Request
        ServerHttpRequest request = exchange.getRequest();
        //获取参数
        MultiValueMap<String, String> queryParams = request.getQueryParams();
        //获取key为token的第一个值
        String token = queryParams.getFirst("token");
        System.out.println("token:"+token);
        //获取所有key为token的值
        List<String> token1 = queryParams.get("token");
        System.out.println("tokens:"+token1);

        if ("admin".equals(token)){
            //进行放行，让它进行执行接下来的操作
            return chain.filter(exchange);
        }

        //不进行放行

        //设置状态码，不让拦截后响应太僵硬
        exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
        //拦截请求,不让它继续以后的操作
        return exchange.getResponse().setComplete();
    }

    //设置过滤器优先级
    @Override
    public int getOrder() {
        return -1;
    }
}

过滤器执行顺序：

每一个过滤器都必须指定一个int类型的order值，order值越小，优先级越高，执行顺序越靠前。

GlobalFilter通过实现Ordered接口，或者添加@Order注解来指定order值，由我们自己指定路由过滤器和defaultFilter的order由Spring指定，默认是按照声明顺序从1递增。

当过滤器的order值一样时，会按照defaultFilter >路由过滤器>GlobalFilter的顺序执行。
